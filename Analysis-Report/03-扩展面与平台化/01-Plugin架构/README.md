# 第三部分：扩展面与平台化

## 第一章：Plugin 架构

### 8.1 为什么 Hermes 的插件面是 fork 关键
对 fork 维护者来说，插件化意味着：
- 私有能力可快速接入。
- 与 upstream 的冲突面显著减小。
- 版本升级时 rebase 成本更低。

### 8.2 三类核心插件面
1. 通用插件：注册 hook、工具、CLI 子命令。
2. 记忆插件：实现 MemoryProvider。
3. 模型提供方插件：注册 ProviderProfile。

### 8.3 hook 的治理价值
pre/post tool、pre/post llm 等 hook 让你可以：
- 做审计与策略拦截。
- 做成本/时延观测。
- 做结果后处理与规范化。

### 8.4 设计原则（非常重要）
项目长期原则是：**不要为单插件把逻辑硬编码进核心文件**。若插件能力不足，应扩展通用插件接口，而非加特例分支。

### 8.5 推荐的 fork 扩展流程
1. 先做插件原型（最小闭环）。
2. 通过 hook 观察真实流量与失败模式。
3. 只有在多个插件共性需求稳定后，再考虑提炼到 core。
